## Part 1: `:workflow improve` Command

### What was built

Both `agent-harness.mjs` and `agent-harness.html` now support:

```sh
aaron workflow improve <name> "feedback"
# or in REPL:
:workflow improve <name> <feedback>
```

The improve command reads the existing workflow JSON, sends it to the agent with the feedback, and writes back a revised definition. It does **not** re-execute the steps — it only revises the step prompts.

### Prompts used

**REPL improve prompt (sent to agent):**
```
Read /workflows/<name>.json — this is the current workflow definition.
User feedback: <feedback>
Revise the workflow definition to address this feedback:
- Update step prompts to be more specific and produce complete, concrete deliverables
- If the goal involves a web page, ensure one step writes a complete self-contained HTML file
  (embed CSS and JavaScript inside the HTML — do not create separate .css / .js files)
- Each step prompt must include the exact output path under /artifacts/<name>/<file>
- Add new steps if needed, remove unnecessary ones, keep 3-6 steps total
Write the revised version back to /workflows/<name>.json.
Do NOT execute the steps — only revise the workflow definition file.
context.emit({ type: 'done', message: 'Workflow "<name>" improved. Run it with :workflow <name>' })
```

## Part 2: Workflow Composition — Option C (No Runner Changes)

### The question

> Do we have examples of workflows that can call skills and other workflows? Will this work?

### Three approaches considered

| Option | Approach | Status |
|--------|----------|--------|
| A | Template expansion `${/path/to/file}` in prompt strings | Not implemented |
| B | `include: [paths]` array field on step schema | Not implemented |
| **C** | Agent reads workflow/skill files via `context.vfs.read()` — no runner changes | **Chosen — proven viable** |
| Embedded JS | Executable code in workflow JSON | Rejected — mixes data/code |

### Why Option C

Option C uses the fact that the agent already has full VFS access. A step prompt can instruct the agent to `context.vfs.read('/workflows/foo.json')`, parse the steps, and use them as a guide. No new keywords, no schema changes, no runner changes.

### Limitations of Option C

- The agent can **read** another workflow and use it as architectural reference (works great)
- The agent **cannot call the LLM recursively** from within `execute()` — so it can't literally re-run another workflow turn-by-turn from code
- Very large single steps (e.g., "execute two full workflows in one turn") may produce incomplete output — agent writes some files but not all before the code block ends

### Proven use cases

#### `csvtools` workflow — Architecture reference ✅
Step 1 prompt (key excerpt):
```
Read /workflows/texttools.json to understand how the texttools CLI was structured.
Use the EXACT same architectural pattern to build a CSV processing CLI at /artifacts/csvtools/.
```

**Result:** Agent read `texttools.json`, parsed the architecture, built matching 3-file project. CLI fully functional:
```sh
printf 'name,age\nAlice,30\nBob,25\n' | node src/index.js count   # → Rows: 2, Columns: 2
printf '...' | node src/index.js filter name=Alice                 # → Alice row
printf '...' | node src/index.js stats                             # → min/max/avg per numeric column
```

#### `devkit` workflow — Sub-workflow execution ⚠️ (partial without continuation pass)
Step 1 prompt (key excerpt):
```
Read /workflows/texttools.json and /workflows/gitreport.json — both are workflow definitions
with a 'steps' array. Execute BOTH workflows' steps yourself, producing all outputs under
/artifacts/devkit/texttools/ and /artifacts/devkit/gitreport/
```

**Result without continuation pass:** Agent wrote `index.html` and one `commands.js` — missed 5 files.
**Result with continuation pass:** All 7 files produced (see Part 3).

### Option C pattern that works well

```json
{
  "id": "build-cli",
  "prompt": "Read /workflows/texttools.json to understand the architecture.
             Build a similar CLI for CSV processing following the same pattern."
}
```

```json

  "prompt": "Read /workflows/texttools.json AND /workflows/gitreport.json.
             Execute ALL steps from both workflows..."
}
```

## Part 3: Continuation Pass — Per-Step Verification Turn

### The problem

The workflow runner accepted a step as complete even when the agent produced partial output. The agent ran, wrote some files, and returned success — the runner had no way to know outputs were missing.

### The fix

After each step's main `runTurn`, run a second verification turn:

```js
// Continuation pass: verify step completion and fill any gaps
output.write(c('gray', `  ↺ [${step.id}] verifying...\n`));
await runTurn(
  `You just ran step "${step.id}". Its task was:\n${step.prompt}\n\n` +
  `Call context.vfs.list() and verify every file or output the step required now exists. ` +
  `If anything is missing or incomplete, write it now. ` +
  `context.emit({ type: 'done', message: 'Step ${step.id} verified' })`,
  state, deps
);
```

Added in both:
- REPL workflow runner (inside `repl()` function, ~line 780)
- CLI `workflowRun()` function (~line 1139)

### How it works

1. Main turn runs — writes most outputs
2. `↺ [step-id] verifying...` appears in terminal
3. Agent calls `context.vfs.list()`, compares against what the step required
4. If gaps found: writes missing files, commits
5. If complete: emits `done` immediately (cheap turn)

### Proof — devkit re-run with continuation pass

The continuation turn's output showed the agent self-correcting:
```
↺ [execute-sub-workflows] verifying...
  Creating missing texttools files...
    file_write /artifacts/devkit/texttools/package.json
    file_write /artifacts/devkit/texttools/src/index.js
    file_write /artifacts/devkit/texttools/src/commands.js
  Creating missing gitreport files...
    file_write /artifacts/devkit/gitreport/cli.js
    file_write /artifacts/devkit/gitreport/report-generator.js
    file_write /artifacts/devkit/gitreport/index.js
  Step execute-sub-workflows verified: All 6 required files created and present
```

## Key Architectural Decisions Made

### Workflow composition strategy: Option C

Use step prompts that reference other workflow/skill VFS paths directly. The agent reads them via `context.vfs.read()`. No new schema fields needed. Trade-off: very large composite steps may need the continuation pass to catch gaps.

### Continuation pass is always-on

Every step gets a verification turn regardless of apparent success. The cost for a "happy path" step is one short LLM turn where the agent confirms completion. The benefit for partial steps can be significant. This is preferable to heuristics (file count thresholds, etc.) because it uses the LLM's own judgment.

### Tightened workflow create/improve prompts

Both harnesses now enforce these rules in the create/improve prompts:
- Steps must name **exact output paths** (`/artifacts/<name>/<file>`)
- Web output must be **self-contained HTML** (CSS + JS embedded, no separate files)
- 3–6 steps total
